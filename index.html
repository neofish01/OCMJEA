1. Assumptions

  • Customer do buying or selling must choose an account first.
  • Every holding will contains only one asset and one cash.
  • The cash is this asset's historical buying price .
  • Holdings can only have cash and no any asset.
  • Customer can pay as any currency, Retire Early will transfer to USD, so I keep rate and currency in table.
  • It's not mentioned that Retire Early provide all kind of stocks, mutual funds and bonds or just one product every asset, so I assume this system will provide all kind of products and use Value Object to keep every kind of any asset.
  • Merchant Bank will handle transfer and validate if lock-up period is over or not.

2. Brief System Overview

  • Retire Early system was designed as a Java Enterprise application accessible by customer through web browser.
  • System was designed to be highly available across North America, Europe, Asia with data centres located on those three continents.
    • Provided design allows it to work on 'commodity' computers, not needing any expensive infrastructure.
  • JMS server is standalone to retrieve the lates price of all products from PriceRight. Even one server is down, the other will take over immediately.
  • Use Primefaces as JSF implementation. The AccountManagerBean calculate the portfolio breakdown and use tagLib to draw charts like pie/ bar chartâ€¦etc.
  • To provide high availability, Retire Early adapts web server cluster, application server cluster and data base cluster. Any parts can be scale up vertically by higher performance hardware. Use DNS as load balancer to make sure load balance. The transaction will be done within 5 seconds, and all account details will be shown with 8 seconds.
  • Customers must use HTTPS protocol to access Retire Early, and the browser must support 128-Bit Encryption or higher..

    7. Risks and Mitigations

    • More customer can join Retire Early.
      Mitigation: Provided architecture allows both vertical as well as horizontal scaling. In case of experienced slowness in clustered database due to increased number of active users. To separate I/O DB is a good solution.
    • Customer cannot do transaction across different continents.
      Mitigation: Because of every continent has its own hardware and database is standalone. If Retire Early want to share customer data, then doing batch job to synchronize data at off-peak time is a good idea.
    • Security issues. 1.Man in the middle attacks 2.Phising 3.DNS spoofing
      Mitigation: Using mutual SSL for external communication. Encrypts and authenticates both parties.Retire Early certificate must be granted by a certification authority
